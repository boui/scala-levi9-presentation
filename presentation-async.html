Introduction

First of all

Concurrency

What does concurrent code consist of?

Consumers/workers/threads/processes doing some work silmultaniously

If you have a bad luck, you have shared mutable state.

It hurts.

Classical concurrency is low level.

What will you experience writing classy concurrent code?(short list)

  • Deadlocks
  • Untraceable or nearly untraceable behavior
  • Bugs is much more difficult to reproduce
  • Threads starvation and bottlenecks
  • Memory usage issues, like cache miss
  • Inconsistent changes, when you forget to restict resource access, phantom reads

Can you avoid writing concurrent code?

No and deal with it

Can you do it fancy with less hardcore?

What are we dealing with, just to start

Process, thread

Managed by OS which means Kernel.

Threads has their own memory and independent flows of control. Within a process.

The process "owns" all the resources like memory, file handles, sockets, device handles, etc.

Timeslicing and context switch

You need to think of a great effort system does to switch context when time slice ends

This things are physical

Green threads

Not scheduled by OS, instead managed by Virtual Machine

Used memory is declared by VM

Lots of languagess support them - Java, Go, Python, Haskell, Rust, Erlang(green processes), Smalltalk, Lua

Locks, semaphores

Low level, and you know it already

Futures

Future is a function executed in another thread which results with something

It may be either success or an error

Success means future return a value, error may be exception, or bussiness logic fail.

And also we have something builded on top of it

Actor

Actors are based on green threads, so actors are lightweight.

What makes thing an actor:

Inner state

Messages queue

Message handling code

Actors can be used to control complex asynchronous flows within messages

Actors are good if you have a state, that needs to be managed

Actors are good for distributed systems

Actors can produce actors hierarchies

Actors are an explicit way to manage mutable state

Rules of an actor

Never do a timeconsuming tasks

Messages are immutable

Avoid system starvation(throttling)

Scala rules of an actor

Use futures for timeconsuming tasks

Use actor states to handle errors

Don't forget that 'self' is a method

Manage throttling carefully, since it might happen

Manage resource usage
